# model + effort routing

## Objective

**Bind a model + reasoning-effort *tier* to each persona, and apply it automatically when that persona is adopted.** Research then runs on the strongest model at the deepest setting, a long build stretch runs cheap, and nobody hand-sets `/model` per session. It is the concrete realization of foundation principle **P9 ("match the model to the work")** and of the **`efficient`** opinion's named *model-routing* lever. It is **not a plugin**: it is a policy (agentm), a per-persona binding (the persona manifest), and an enforcement surface (crickets agent-defs + the host).

## Overview

Routing collapses two dimensions — **model strength × reasoning effort** — into one ordered **tier scale**, because in practice they co-vary (harder work wants a stronger model *and* more thinking). A persona declares a tier; adoption applies it. Frontier reasoning concentrates into bounded planning/verdict sessions while execution runs cheap everywhere else; the tier scale routes both sides of that split.

### The tier scale

| Tier | The work it fits | Effort |
|---|---|---|
| **T4 · Deep** | research · adversarial audit · the hardest reasoning | `max` + orchestration |
| **T3 · Architect** | cross-system design · roadmap · priority / board calls | `max` |
| **T2 · Author** | planning · PM · design authoring · holding the technical bar | `high` |
| **T1 · Execute** | long autonomous build stretches | `medium` |
| **T0 · Mechanical** | rote edits · log-scraping · deterministic passes | `low` |

*Concrete model ids per tier — Claude and Gemini/Antigravity alike — live in [token-audit](https://github.com/alexherrero/crickets/wiki/crickets-token-audit)'s versioned routing table, the single home, refreshed by `content-refresh` in step with `pricing.py`. This chart states the portable tier + effort structure only; it does not need an edit when a model is renamed or replaced. The Claude-side table is the one built first — Gemini/Antigravity rows lag until that side of the portfolio needs its own consumption path (named honestly, not claimed as symmetric on day one). The effort ladder is `low · medium · high · max`; `ultracode` is the Claude-Code top-tier **orchestration mode**, not a fifth effort level.*

*T0/T1 rows carry the **advisor rider** (see Escalation below): `opusplan` stays T1's default pending session-cost data — the flip to a cheap main loop + advisor is that capture's first named consumer, not a decision made here.*

### Persona → tier

| Persona | Tier | Note |
|---|---|---|
| Researcher · Reviewer (audit) | **T4** | the hardest reasoning + adversarial breadth |
| Architect · Planner *(roadmap mode)* · Troubleshooter | **T3** | cross-system + priority calls |
| Designer · Tech-Lead · Planner *(planning/PM mode)* | **T2** | authoring + the technical bar |
| Engineer / Worker | **T1** | the long build stretch (`opusplan`) |
| Operator · Maintainer · Memory | **T0–T1** | read/report + mechanical repair |

*A persona's tier is its **default**; a mode-dependent persona escalates one tier for a genuinely harder sub-task (e.g. Planner → T3 for a roadmap call). The escalation is bounded by the persona's declared ceiling.*

*T2 · Author is not one model: the work-type table (token-audit) splits it by **shape** — roadmap/architecture calls route to Opus 4.8 at high effort, transcription-shaped authoring (staging a plan whose reasoning is already settled elsewhere) routes to Sonnet 5 at high effort. The persona→tier map above states the tier; the work-type table states which of T2's shapes applies.*

## Design

### The policy — one tier scale, owned by `efficient`

The scale is the canonical artifact: each tier is a (model-class, effort) bundle named by intent. It realizes the **`efficient`** opinion's *model-routing* lever — `efficient` is "cheap as the job allows, above the quality floor," and the scale is how that judgment becomes a concrete pick. It implements foundation **P9**. The scale lives here so it is defined once; personas, the opinion, and the enforcement surface all reference it rather than restating model names.

### The known models, and the classifier that resolves to them

The scale currently draws on five pinned model ids (`pricing.py`, the ground truth): `claude-opus-4-8`, `claude-sonnet-5` (current generation, introductory pricing through 2026-08-31), `claude-sonnet-4-6` (prior generation — re-pinned off the three agent-defs that cited it as of the Wave D personas build, 2026-07-06; see Amendment log), `claude-haiku-4-5`, and `claude-fable-5`. Fable is real and pinned, but at roughly 2× Opus's per-MTok rate it is the single most expensive row token-audit's table carries, and no work-type resolves to it today — the standing operator convention keeps it unused, and the Mythos incident (112 sub-agents silently inheriting it) is the concrete reason why: an unrouted, expensive model is exactly what a silent inheritance defaults to if nothing positively identifies the work first.

That positive identification is **the classifier** (`classify_work_type`, hosted in [token-audit](https://github.com/alexherrero/crickets/wiki/crickets-token-audit) alongside its table): given a dispatch or a staged plan, it resolves to a work-type key — from a persona's own declared tier, from an ad-hoc role-name match, or, failing both, an explicit `UNCLASSIFIED-DEFAULT` rather than a silent guess. The tier scale above is the policy the classifier's output is checked against; the classifier is how "which tier does *this* dispatch belong to" stops being answered ad hoc, in-prompt, per session.

### The binding — a tier axis on the persona manifest

The persona manifest gains a **`tier:`** field (and, where a persona spans modes, a `tier-ceiling:` for bounded escalation). This is the net-new axis: today the manifest declares stance · `enhances:` · `requires:` · opinions · modes · triggers, and **carries no model or effort at all**. Adding `tier:` makes "which model this stance runs on" a declared, validated property (extending `check-personas.py`), not an operator habit. The persona roster gains the tier column shown above.

### Enforcement — four surfaces, by what the host can enforce

A tier becomes a real model+effort differently depending on how the persona is running:

- **Spawned sub-agent personas → automatic, confirmed live.** The agent-def carries `model:` (and the new `effort:`) frontmatter as a fallback default; the Agent tool's per-invocation `model` param (enum `sonnet`/`opus`/`haiku`/`fable`) overrides it and is the real, host-documented lever — confirmed live in P12 (`PLAN-efficiency-dispatch` Task 8): an `Explore` dispatch passed `model="sonnet"` self-reported back `claude-sonnet-5`. Per-dispatch **effort** stays unbound: the Agent tool has no per-invocation effort param, and whether agent-def `effort:` frontmatter itself binds is unverified across a session boundary — Task 2b's live spike hit an unrelated blocker (a new agent-def dropped into `.claude/agents/` isn't hot-reloaded mid-session, so the spike couldn't complete without ending the session). *Re-audit trigger:* re-run the agent-def `effort:` spike in a fresh session — if it binds, the roster gains a real effort axis.
- **Phase-command turns → advisory only, via the prose nudge — command frontmatter is decorative (retired 2026-07-04).** P12 Task 0's live spike found `model:`/`effort:` frontmatter on a `kind: command` skill/command has no observable effect on the turn's model: three throwaway scratch commands pinned to `claude-opus-4-8` / `opusplan` / `claude-sonnet-5` all reported back the session's own baseline model, unchanged. A command injects text into the turn already running at whatever model the client picked at turn start — it is not a new dispatch the way the Agent tool is. The mechanism actually doing work is the prose nudge (`> **Recommended model for this phase:**`) already carried alongside the frontmatter in `work.md`/`plan.md`/etc.
- **Non-command turns and cross-turn persistence → advisory.** Same nudge, same limit — free-form conversation turns and anything that must persist across turns surface the tier as a nudge ("recommended: T2 · Author — Opus high; `/model` to override") rather than an enforced switch, since the host has no lever to bind a model beyond a single dispatched sub-agent call.
- **Build default → `opusplan`, session-level only.** The global `opusplan` alias (Opus plans, Sonnet executes) is the host-level realization of **T1 · Execute** for build sessions — a built-in two-tier split that covers the worker's plan/execute rhythm without per-step selection. It is never a legal value for the Agent tool's per-invocation `model` param and has no meaning for a single dispatched turn; the single-implementer invariant means it's never needed on the dispatch path.

### Escalation — the reverse path

Enforcement above routes downward — matching the model to the work at adoption or dispatch. The reverse direction, a cheap session hitting work above its tier, has three answers rather than a self-switch (the host permits neither a hook nor a running turn to change its own model):

1. **The advisor rider.** T0/T1 sessions set `advisorModel` so a stronger model is consultable mid-task without switching the session's own model — config-only, no build. Anthropic-API-only; pinned to Opus 4.8 for now (Fable-as-advisor is version-gated to 2.1.170+, and the advisor must be at least as strong as the main session).
2. **The `/work` tripwire.** The same verification gate failing three consecutive attempts on one task emits a handoff-pack escalation entry (packed context + recommended tier), announces loudly, and stops at the safety gate — it never switches the model itself, because it can't. This is insurance for unattended sessions rather than a claimed workload: the failure mode it guards is a cheap model grinding a failing gate, with test-weakening as the worst exit.
3. **The budgeted-oracle primitive, held behind evidence.** A budgeted frontier-tier subagent dispatched only for bounded hard-reasoning steps exists host-natively via Workflow's per-call `model`/`effort` plus its token `budget` — named here as the reach-for shape, not built now. Its trigger is the first tier-attributable correction observed in a cheap session; until then, building it would duplicate the advisor + `opusplan` against no observed demand, on a convention-grade budget the Mythos incident already showed to be insufficient.

### Cross-host — Claude and Gemini

The tier is the portable unit; each host renders it through the chart. Claude Code renders via `model:` + effort + (at T4) `ultracode`; Antigravity/Gemini render via the Pro/Flash classes + a thinking budget. Where a host can't honor a dimension (e.g. an effort field it doesn't read), the tier degrades to the nearest model-only pick and the advisory nudge carries the rest — the same graceful-skip posture the rest of the portfolio uses.

### How token-audit feeds it

[token-audit](https://github.com/alexherrero/crickets/wiki/crickets-token-audit) is measurement-only and one-way: its designed auto-capture (a per-model `session-cost` record reviewed in the **dreaming** pass) surfaces when a persona is running hotter than its tier warrants — the longitudinal signal that **tunes** the scale and the persona→tier map over time. token-audit's newly designed routing-conformance report reads its own hosted table from the other side — grading model-used vs. work-type after the fact — but that is reporting, not routing: routing decides, token-audit only ever measures and reports. Routing decides; token-audit never picks a model.

## Risks & open questions

- **The whole thing is designed, not built** (`[PENDING-IMPL]`) — only the model-only, three-role as-built slice exists in crickets; the effort axis, the manifest field, the roster-wide binding, and the activation that reads it are greenfield (the persona "activation plumbing" is itself the unbuilt core).
- **In-session enforcement is narrower than earlier believed — command frontmatter is decorative, not enforced (retired 2026-07-04).** P12 Task 0's live spike found `model:`/`effort:` frontmatter on a `kind: command` skill/command has no observable effect on the turn's model — a command injects text into the session's already-running turn, it doesn't open a new dispatch. The one confirmed-live enforcement mechanism is the Agent tool's per-invocation `model` param on a genuine sub-agent dispatch (Task 8: `Explore` dispatched with `model="sonnet"` self-reported `claude-sonnet-5`). Phase-command turns and everything else stay advisory-only via the prose nudge, unchanged from the original design intent. *Re-audit trigger:* a future CLI version demonstrably honoring command-level frontmatter.
- **The Agent tool has no per-invocation `effort`.** Only `model` can be passed per dispatch; whether agent-def `effort:` frontmatter itself binds is unverified across a session boundary — P12 Task 2b's live spike couldn't complete (a new agent-def isn't hot-reloaded mid-session) — so effort stays out of this design's reachable dispatch-time mechanism until a fresh-session re-spike confirms or denies the frontmatter fallback.
- **Version-gate hazards** — the installed CLI (2.1.112) trails the documented version (2.1.201): **Explore inherits the session model starting at 2.1.198** (the cheap-floor-Explore assumption silently inverts on upgrade — the announcement contract must cover built-in-agent inheritance, not just explicit dispatches). Confirmed live below that threshold (P12 Task 8): a model-less `Explore` dispatch self-reported `claude-haiku-4-5-20251001`, not the session's own `claude-opus-4-8` — Explore carries its own fixed default, it does not inherit. The `INHERITED` announcement label conservatively covers both "agent's own default" and "true inheritance," since the dispatch site can't distinguish them without introspecting the target agent-def; re-audit once the installed CLI reaches ≥2.1.198. Fable-as-advisor is gated to 2.1.170+.
- **Effort-field host support varies** — Claude Code's effort/ultracode and Gemini's thinking budget are not the same control; the tier degrades gracefully where a dimension isn't honored.
- **Hosting the model-id table in token-audit (not here) trades a policy/data split for a single source of truth.** An operator reading this chart sees the tier structure only, not which model is actually running for a work-type; they must follow the pointer to token-audit. The alternative — keep ids in this chart — was rejected because it duplicates `pricing.py`'s spelling responsibility, the exact drift the one-pinned-source pattern exists to prevent.
- **Quality-shaped escalation has no deterministic trigger.** The Escalation section's `/work` tripwire (above) catches a session **stuck** on a failing gate — a counter on verification hooks, mechanical. It does not catch a session that is **confidently wrong**: cheap-tier output that passes every gate but is substantively wrong because the work genuinely needed a stronger model's judgment. There is no deterministic signal for "this session is out of its depth" the way a failing test signals "this session is stuck." The residual coverage is judgment-triggered, not mechanism-triggered: the advisor rider, if the session recognizes its own uncertainty and calls it, or a human noticing later (at `/review`, or reading the output). Named honestly rather than implying the Escalation section is a complete answer to "a cheap session hits work above its tier" — it fully answers the *stuck* sub-case; the *quietly wrong* sub-case stays advisor/human territory.
- **The chart decays on model releases** — now token-audit's table decays instead; **`content-refresh` carries the re-pin as a named checklist item** (mechanical re-pin auto-applied; a new-model *tier* placement surfaces for review), triggered by the weekly **CI model-drift detector**, and a model bump is also a P8 re-audit trigger.
- **Re-audit triggers:** add the `tier:` manifest field + validator; generalize the agent-def routing + add `effort:`; wire activation to apply the tier at adoption; wire `content-refresh` to re-pin token-audit's table; re-pin on every model release; confirm the persona→tier map against real `session-cost` trends once token-audit's capture ships; Gemini/Antigravity rows land in token-audit's table when that side needs its own consumption path; a host ever exposes a self-assessed-uncertainty signal a quality-escalation trigger could key on.
