# AgentM Experience & Dreaming

## Objective

Experience is **how agentm gets better over time** — it turns the agent's own past sessions, and the wider world, into sharper judgment, so the next session starts further ahead than the last. It is what makes the memory *grow* rather than just persist.

## Overview

Experience runs in **two directions**:

- **Backward** — learning from its own past. Every finished session leaves something behind that sharpens the next. **Built today.**
- **Forward** — learning from the world. On a schedule, when configured, the agent reaches approved sources, brings back ideas to improve itself, and surfaces them to you. **Thin v1 built, opt-in.**

## Design

### Backward experience

- **Reflection** *(built)*. After a session ends, a pass mines the transcript for durable entries, each **classified by `kind`** (preference · workflow · fix · …; see the [Memory System](agentm-memory-system.md) for the taxonomy). It routes each candidate by confidence into **three lanes**: **HIGH** → auto-saved; **MEDIUM** → the inbox or surfaced for review; **LOW** → the inbox to triage later. The routing *mode* is set by `MEMORY_REVIEW_MODE` (auto / silent / interactive); the confidence tiers themselves are hardcoded, not a tunable threshold. A Stop-event hook runs it at session end; an idle hook catches sessions that crashed before it could fire. Reflection also seeds **idea incubation** (a skeleton an idea-researcher sub-agent later fills).
- **Heat-based curation** *(built)*. Which memories load *every* session changes with use: a heat policy promotes an on-demand entry into the always-load set once it's been hit enough times across enough distinct sessions, and demotes cold ones — down to a safety floor, leaving pinned entries alone. Built and tested; it keeps the always-load set earning its token cost.
- **Crystallization** *(designed and built, schema + primitive only, no automatic trigger)*. Distils an exploration into a structured digest — question · investigation · findings · lessons · open threads — instead of leaving raw transcript fragments behind. It's the phase-close counterpart to dreaming's whole-corpus pass. **`harness/skills/memory/scripts/crystallize.py`** (built 2026-07-07, `PLAN-wave-e-experience` task 2) ships the five-field digest schema and its write/read primitives — `crystallize_exploration()` writes a `kind: crystallized` entry through the existing `save_entry()` (no new store), `parse_digest()` reads one back and raises on a missing section. It's **operator-invoked**, not wired to any phase boundary: "the close of a completed exploration" names no bounded, detectable trigger anywhere in this codebase today (no hook, marker, or session type of that name exists), so automatic phase-close triggering is deferred as a v2 concern — the same thin-manual-path-first precedent dreaming (`/dream`) and forward learning (its single deterministic pass) already set in this file. Distillation itself — turning a raw transcript into the five field values — is also out of scope here: `crystallize_exploration()` takes an already-composed `CrystallizationDigest`, it doesn't mine a transcript. **`[PENDING-IMPL]`** — the phase-close trigger (needs coordination with the crickets developer-workflows phase loop or a new session-boundary marker) and the distillation step that produces the five fields.

### Forward experience — designed and built (thin v1, opt-in)

Backward learning can only sharpen what the agent has already seen; forward learning brings in what it *hasn't* — a periodic, opt-in pass that reaches approved sources and surfaces what's worth knowing.

- **Approved-source learning.** On a schedule (operator-configured), the agent pulls from a list of **approved sources** — feeds, named repositories, the web — and mines them for ideas that could improve it: techniques, tools, patterns, conventions. What it finds is **surfaced to you** to accept or pass on; nothing is adopted silently. The source list and the cadence are operator-controlled — opt-in by design.
- **The approved-source pipeline exists now:** `harness/skills/memory/scripts/forward_learning.py` (built 2026-07-07, `PLAN-wave-e-experience` task 1) generalizes the import watchlist's shape — find → screen → surface — beyond skills to ideas, patterns, and references. Sources are read from `_meta/forward-learning-sources.json`, which is **absent by default**: no config means a scan finds nothing, so the pipeline is inert until an operator opts in by naming sources. Unlike the import watchlist below, this v1 is a **single deterministic pass** — one rubric (score thresholds reused from `adapt_skills.py`: ≥3 HIGH, ≥1 MEDIUM, else LOW) does find, screen, and classify together, with no `adapt-evaluator`-style semantic judge sub-agent. A Pass-2 LLM judge is a natural v2 graduation, not built here. MEDIUM/HIGH candidates land in a new, generalized `personal/_watchlist/` (sibling to the skill-specific `personal/_skill-watchlist/`); LOW is dropped, never written. Per-source watermarks live in `_meta/forward-learning-cache/`. Nothing outside those two locations is ever touched — no auto-adoption anywhere else.
- **The import watchlist (adapt-don't-import)** was the one piece that existed before this task, and still owns *skills* specifically: a deterministic rubric (`adapt_skills.py`) enriches candidate external skills, a judge sub-agent (`adapt-evaluator`) classifies them HIGH/MEDIUM/LOW, and a review CLI (`watchlist_review.py`) lets you promote / dismiss / defer. `watchlist_review.py` is now generalized (same task) so the same CLI reviews both `_skill-watchlist/` and the new `_watchlist/` as one merged surface — including a fix so `dismiss_entry` archives each entry back into whichever root it actually came from, rather than always the skill-watchlist archive.
- **The runner (the background-job executor) — built 2026-07-06.** Backward learning rides session events; forward learning has no session to ride, so it runs on the **runner** — fired by the host's scheduler (see the [Runner design](agentm-runner.md)). `templates/jobs/forward-learning.yaml` (built 2026-07-07) registers the pipeline as a `schedule: daily`, `dry_run: true` job manifest, opt-in like [dreaming's job template](agentm-runner.md) — no `budget:`, since the thin v1 pass makes zero LLM calls. The runner is also the home for other periodic upkeep (dreaming, index maintenance); it's a shared substrate primitive the Maintainer and Researcher persona loops wait on too.

**Designed and built, thin v1, opt-in.** The approved-source pipeline is real and tested (20 tests across `test_forward_learning.py`, `test_watchlist_review.py`, `test_forward_learning_job.py`), but it isn't the full design: no sources are configured by default, screening is one deterministic rubric rather than the two-pass deterministic-rubric-then-semantic-judge split the import watchlist itself uses, and the job manifest ships `dry_run: true` requiring manual `.harness/jobs/` install like every other runner job template. *Re-audit trigger:* revisit this note if a Pass-2 semantic judge is built, or once real approved sources are configured and calibration evidence justifies flipping `dry_run` to `false`.

### Dreaming — designed and built (dry-run / opt-in)

Memory's "sleep": a whole-corpus consolidation pass — dedup, contradiction triage, compression, insight-generation — producing a derived insights layer. The design is locked. Its prerequisite, the revert-log primitive, exists (`harness/skills/memory/scripts/revert_log.py`, built 2026-07-07), and the pipeline itself now has a thin, operator-invoked implementation: `harness/skills/memory/scripts/dream.py` (`run_dream(vault_path, *, run_id=None) -> DreamDigest`, built 2026-07-07). It runs dedup (stdlib `difflib`, threshold 0.92), contradiction triage (same-`slug`-differing-body, advisory only), and compression (`supersedes:`-chain compaction, never deleting a source) as **proposal-only** stages — no source file is mutated by a run — and writes every disposition to a staged digest (`_dream-staging/<run_id>/digest.md`) plus a machine-readable manifest (`_dream-staging/<run_id>/proposals.json`) carrying a prospective revert pointer for the confirm step to apply. Insight-generation is the one stage that writes directly, always as an additive `status: candidate` file under `_dream/insights/<run_id>.md`, per the design's own carve-out for non-source-touching writes.

The confirm step now exists too: `harness/skills/memory/scripts/dream_confirm.py` (`list_pending`, `confirm`, `expire_stale`, built 2026-07-07) is the only path a staged proposal ever applies through — `confirm()` calls `revert_log.RevertLog.record_and_apply(...)` directly against the manifest's structured mutations, never a source-file write of its own. An unconfirmed proposal carries a TTL (default 30 days, a calibration-era judgment call, not a locked design number); past TTL, `confirm()` raises rather than silently applying, and a separate `expire_stale()` bulk pass marks stale proposals expired without touching source files. Confirmation state lives in a sidecar (`_dream-staging/<run_id>/state.json`) next to the staging directory, not in the revert-log journal itself. With this, the **manual path — stage → confirm → apply-through-revert-log — is end-to-end complete and fully revertible**: an operator can run `/dream` by hand today and get the full pipeline the design's Objective describes.

The scheduled half now exists too: `templates/jobs/dream.yaml` (built 2026-07-07) registers the whole-corpus pass as a job manifest on the [Runner](agentm-runner.md) — `command:` wires `dream.py`'s CLI, `schedule: weekly` + `lookback: 14d` stand in for the design's "activity-gated" framing until the runner's `gate:` field graduates from parsed-but-unevaluated to real (`scripts/runner/cycle.py`, as-built), and `tier: T3` reflects that insight-candidate writes are new agent-generated content rather than operator-curated T2. It carries no `budget:` — the thin v1 pipeline makes zero LLM calls. Two things keep this from being a plain "ship it and forget it" flip. First, it ships **`dry_run: true`**: per the plan's own Constraints, a due cycle reports `ran=False, dry_run=True` and proves the command never executes, and promoting to live-apply is a separate, explicit future decision, not a consequence of the manifest merely existing. Second, `.harness/` is entirely gitignored in this repo, so the manifest can't live at a tracked path there — it ships as a tracked template at `templates/jobs/dream.yaml`, following the same `templates/` → `.harness/` copy precedent as `.harness/hooks/*` and `.harness/verify.sh`, and a repo opts in by copying it to `.harness/jobs/dream.yaml` by hand; `install.sh` has no automatic seeding step for runner jobs yet (this is the first job manifest shipped at all, ahead of forward-learning's or health-check's). Tested by `scripts/test_dream_job.py` (3 passing tests): the manifest parses against `scripts/runner/manifest.py`'s schema; a due cycle against the shipped template never touches `_dream-staging/`; and a cycle with `dry_run` overridden to `False` for the test only (the shipped template itself stays `dry_run: true`) produces a digest structurally identical to task 2's direct `dream.run_dream()` call.

**Designed and built, shipped dry-run / opt-in.** All four tasks are built and tested — the pipeline, the revert-log safety net, the confirm/expire flow, and now the scheduled job manifest — so this is no longer a bare `[PENDING-IMPL]` gap. But it isn't a plain as-built either: the manifest is a template a repo must deliberately copy into `.harness/jobs/`, not something `install.sh` seeds on its own, and even once registered it stays `dry_run: true` — proposals stage to `_dream-staging/` for operator review, nothing auto-applies — until an operator separately promotes it to live. Re-audit this note when `install.sh` gains a seeding step or a repo's calibration evidence is recorded as sufficient to flip `dry_run` to `false`.

### Where the learning lands

What's learned routes into the **general memory** the same way anything else does — as **`kind`-classified entries**: a learned preference, workflow, fix, or domain-reference becomes a typed entry, classified by reflection on the way in (see the [Memory System](agentm-memory-system.md)). Two other destinations sit alongside it:

- **The idea incubator / watchlist** — an ingested idea, article, or candidate skill (forward learning) lands here **for later review**, not adopted silently.
- **The Opinion supplement** — the part that's a *standard* sharpens an opinion's learned layer over time (the *accumulate* loop — spec landed, implementation deferred; see the section below and the condensed twin in the [Opinions design](agentm-opinions-and-gates.md)).

Everything routes through the existing memory engine — no new store; heat curation runs continuously over the always-load set.

### The Experience → Opinions accumulate loop — spec landed, implementation deferred

**`PLAN-wave-e-experience` task 3, 2026-07-07.** This section lands `ACCUMULATE-LOOP-SPEC-DRAFT.md` (a Fable-authored draft rendered 2026-07-06) as-is — the design amendment for the one pillar-to-pillar flow that previously had no contract: how a signal from Experience's capture paths becomes a lasting sharpening of an Opinion's learned supplement. No code implements any of the four stages below yet; that build is explicitly deferred to a follow-on plan. This section is the contract that build will implement against.

**The one-sentence design:** the accumulate loop is the style-learning loop generalized from voice to standards — the same edit-driven, operator-gated capture the wiki system already proved, with a recurrence gate in front and the coded base as an unoverridable floor behind.

**1 — Route, don't invent.** No new pipeline. The loop is a routing rule inside the capture paths that already exist: when reflection (session-close), edit-driven capture, or the watchlist produces a candidate that is *standard-shaped* — a rule about how work should be judged or done, not a fact or preference — it targets an opinion supplement instead of general memory. Classification is deterministic-first: `kind` + the nine opinion names as anchor vocabulary + rule-shaped verb heuristics; the LLM assists only at MEDIUM confidence, and the tri-lane routing (`MEMORY_REVIEW_MODE`) applies unchanged.

**2 — The signal → opinion map:**

| Signal source | Target opinion |
|---|---|
| `/review` correction patterns; adversarial findings that recur | `good` |
| gate/battery lessons; done-criteria misses; wake-on-PR-checks findings | `done` |
| token-audit routing-conformance findings; tier/cost lessons | `efficient` |
| process/architecture lessons from retros, incidents, design reviews | `how-we-engineer` |
| recoverability incidents (safety-gate stops, near-misses) | `recoverable` |
| privacy/PII incidents and scrubber findings | `private` |
| **voice/prose lessons** | **NOT opinions — stay in the style overlay** (the existing loop; double-capture is forbidden — one lesson, one home) |

**3 — The anti-corruption contract (the heart of it).** Three stacked guards:
- **Recurrence gate:** auto-append to a supplement requires the same lesson surfacing from **two distinct sessions** (the heat-policy discipline applied to standards — one incident is an anecdote, not a standard). Single occurrences park in the opinion's inbox lane for operator triage.
- **Extend, never override:** the fold serves base-then-supplement with the coded base authoritative. A supplement entry that *contradicts* its base is never served silently — it surfaces as a proposed base change, because overriding the shipped standard is a check-in (a human act), not a memory write. This keeps the corruption ceiling at *additive noise*, which the dreaming/consolidation pass (same wave) dedups and compresses.
- **Provenance or it didn't happen:** every supplement entry carries session/commit/incident refs and a supersedes chain. The dashboard's opinions family gains a supplement-health check: percentage of entries with provenance, inbox staleness count — an unhealthy supplement is visible, not discovered.

**4 — Cadence.** Capture is continuous (rides the existing session-close reflection). Promotion to a served supplement is recurrence-gated or operator-confirmed — never time-scheduled. Base check-ins happen at release time by hand. Maintenance (dedup, compression, contradiction sweep) belongs to dreaming's whole-corpus pass — the named owner, same wave, no second mechanism. **Dreaming's task 4 (the scheduled runner job) shipped 2026-07-07** ([PR #244](https://github.com/alexherrero/agentm/pull/244)) — once that job's own dry-run posture is promoted to live, this is the pass that sweeps supplements; no separate consolidation mechanism should ever be built for opinions specifically.

**Why not the alternatives:** a dedicated learning pipeline duplicates three shipped capture paths for zero new signal; LLM-judged auto-promotion violates deterministic-first exactly where corruption matters most; supplement-overrides-base inverts operator-voice-wins and would let two bad sessions rewrite a shipped standard.

**Re-audit triggers (carried forward from the draft):** when dreaming's consolidation ships live (not just dry-run), confirm it actually sweeps supplements — else schedule a lint; if any opinion's inbox lane exceeds ~20 untriaged entries, the recurrence threshold is too loose for that signal source; revisit the extend-never-override ceiling only if a supplement-vs-base contradiction recurs ≥3 times (that's the base being wrong, not the loop).

**`[PENDING-IMPL]`** — none of Stages 1–3's routing rule, signal-map, or anti-corruption guards are built. That implementation is deferred to a follow-on plan; this section is its contract.

## Dependencies

- **Writes to Memory** — every direction lands in the [Memory System](agentm-memory-system.md), and Experience is the **growth engine** that grows it into the interconnected knowledge base (its *How it grows* section).
- **Sharpens Opinions** — the learned supplement is what makes an opinion's vault layer grow over time (the [Opinions design](agentm-opinions-and-gates.md)); the precise *accumulate* loop is designed, not specified.
- **The runner is shared substrate, built 2026-07-06** — the Maintainer + Researcher persona loops ([Personas design](agentm-personas.md)) run on it too, once each of those is built.
- **crickets touch is light** — the memory-engine sub-agents (`memory-idea-researcher`, `adapt-evaluator`) run read-only inside the harness; reflection mines sessions that *used* crickets tools but depends on none; forward learning reaches *out* to sources. Experience keeps working on a bare agentm.
